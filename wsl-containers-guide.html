1 WSL Containers とは

Build 2026 で発表 → 2026年6月29日にパブリックプレビュー開始（WSL 2.9.3 同梱）

WSL Containers (WSLC) は、Windows から Linux コンテナを直接 ビルド／実行／管理するための WSL ネイティブ機能です。これまで必要だった Docker Desktop などのサードパーティ製コンテナランタイムが不要になります。

  • CLI: wslc.exe（Docker 風のコマンド体系。container.exe は wslc.exe を呼ぶ組み込みエイリアス）— アップデート後に PATH へ自動追加
  • ビルド API/SDK: NuGet パッケージ（nuget.org および WSL リリースページで配布）。C / C++ / C# 対応、MSBuild・CMake と統合
  • GPU: 初日から対応（--gpus all）
  • VS Code: Dev Containers 拡張（0.462.0-pre-release 以降）で連携
  • エンタープライズ統制: Intune / GPO によるポリシー、Microsoft Defender for Endpoint による監視（→ §5）
📌
ステータス（2026-06-30 時点）: パブリックプレビュー。pre-release チャネルでのみ提供され、一般提供（GA）は 2026年秋予定。仕様は GA までに変わる可能性があります。

4 仕組み・アーキテクチャの詳細

なぜ Docker Desktop なしで動くのか／virtiofs・consomme とは何か

全体像：WSL の仮想化基盤の上に成り立つ

WSL2 はもともと、Windows 上の 軽量ユーティリティ VM の中で本物の Linux カーネルを動かしています。WSL Containers は、このWSL の仮想化基盤（軽量 VM + Linux カーネル）の上で Linux コンテナを動かす仕組みです。
だからこそ、Docker Desktop のように専用の VM やデーモンスタックを別途持つ必要がないのがポイントで、Windows 側の wslc.exe からこの基盤を直接操作します。

背景 上図は WSL2 と Linux コンテナ技術の一般的な仕組みに基づく概念図です。コンテナ分離の正確な内部トポロジ（共有VMか／コンテナ毎の分離か、namespaces・cgroups の構成等）は公式に開示されていません。 virtiofs・consomme・GPU・SDK・エンタープライズ統合は公式発表で確認済みです。

① virtiofs — 新しい既定ファイルシステム 公式

WSL Containers の既定ファイルシステム。公式は「Windows ファイルアクセスが 2倍速」とうたっています。（背景：virtiofs は VirtIO の共有メモリ転送でファイルアクセスを行い、WSL2 が従来 Windows↔Linux 間で使ってきた 9P（Plan 9） プロトコルより直列化オーバーヘッドが小さい、というのが高速化の一般的な理由です。）

② consomme — 新しい既定ネットワークモード 公式

互換性の向上を狙った既定ネットワークモード（WSL 2.9.3 で旧称 VirtioProxy から改称）。Linux のネットワークトラフィックを Windows 経由でリレーします。これにより Linux アプリが、Windows と同じネットワーク環境・セキュリティポリシー・企業統合（VPN／プロキシ／ファイアウォール 等）の恩恵を受けられます。

③ GPU パススルー 公式

初日から GPU 対応。--gpus all で CUDA などの GPU ワークロード（PyTorch の CUDA 検出例が公式に掲載）を実行できます。WSL がもともと備える GPU パススルー（CUDA on WSL）基盤を活用します。

④ ビルド API / SDK と VS Code 公式

  • WSL Container API: NuGet で提供（nuget.org・WSL リリースページ）。C / C++ / C# からコンテナ操作・ビルドを自動化。MSBuild・CMake と統合。
  • VS Code Dev Containers: 拡張 0.462.0-pre-release 以降で、設定の "Docker Path" を wslc にするだけ。既存の devcontainer.json をそのまま使えます。

⑤ その他の WSL 改善 公式

同リリースでは、メモリ回収（memory reclaim）の改善も入り、使用しなくなったメモリを段階的に Windows ホストへ返却するようになりました（コンテナ専用機能ではなく WSL 全体の改善）。

5 エンタープライズ統合（管理・セキュリティ）

公式ブログ「Integration with enterprise tools」— WSL の管理基盤をコンテナにも拡張

WSL がもともと持つエンタープライズ管理・セキュリティの統合を、コンテナにも広げるのが大きな特徴です。distro とコンテナで一貫した管理・監視カバレッジを狙います。

🛡️ Microsoft Defender for Endpoint でコンテナを監視 公式

既存の Defender for Endpoint プラグイン（for WSL）を更新し、Linux コンテナのイベントも監視。distro でもコンテナでも同等のセキュリティカバレッジを提供します。

ℹ️
このコンテナ監視は現在 private preview（サインアップ制）です。MDE プラグイン自体は Microsoft Intune で展開できます。

⚙️ Intune / GPO でポリシー制御 公式

管理者は次を制御できます：

  • ユーザーが WSL distro / コンテナを使えるかの可否
  • イメージ取得を許可する コンテナレジストリの allowlist（許可リスト）

現時点では GPO / ADMX ポリシーで利用可能。Intune ダッシュボードの公式対応は（公開後）数週間以内に提供予定とされています。

⚠️
MDE プラグイン（for WSL）の既知の制限（Microsoft Learn）: ARM64 プロセッサ非対応／マルチセッション版 Windows 10・11 非対応／カスタムカーネル・カーネルコマンドライン使用時は可視性を保証しない／短命なコンテナ（稼働 ~30分未満）はポータルに現れないことがある／対象は WSL2（WSL1 は無効化推奨）。

必要 WSL
2.9.3+（pre-release）
現在の WSL
2.7.10.0 ⚠️
OS（報道）
Windows 11
CLI
wslc.exe

※ Windows バージョン要件は公式ブログでは明記されていません（各種報道は Windows 11）。この PC の Windows ビルドは 26200 系（Windows 11 相当）です。

⚠️
重要：このセッション内からは完走できません。 手順は wsl --shutdown（＝実行中の WSL を全終了して再起動）を含みます。これはいま動いている WSL セッション自体を終了させるため、必ず Windows 側のターミナル（PowerShell / cmd）から実行してください。
  1. 現状バージョンを確認

    2.9.3 未満なら更新が必要です。

    PowerShell（Windows）
    PS> wsl --version
  2. WSL を pre-release に更新

    wslc.exe を含む新ビルド（2.9.3+）が入り、PATH に自動追加されます。
    ※ この更新コマンド自体は実行中の WSL を止めません（インストールのみ）。

    PowerShell（Windows）
    PS> wsl --update --pre-release

    うまくいかない場合は GitHub Releases（2.9.3） から手動インストールも可能です。

  3. WSL を再起動して新ビルドを反映

    このステップが現セッションを終了させます。 Windows ターミナルから実行し、その後 WSL を開き直してください。

    PowerShell（Windows）
    PS> wsl --shutdown
# この後、WSL ターミナル／ディストロを開き直す
  4. 反映を確認（2.9.3+ になっていること）
    PowerShell（Windows）
    PS> wsl --version
PS> wslc --help     # wslc が使えれば成功
  5. （任意）VS Code 連携

    Dev Containers 拡張を 0.462.0-pre-release 以降に更新し、設定の "Docker Path" を wslc に変更します。JSON のキーは dev.containers.dockerPath（ユーザー設定＝application スコープ。ワークスペース設定では不可）。

    settings.json（User）
    {
  "dev.containers.dockerPath": "wslc"
}

8 動作確認・サンプルコマンド

公式ブログ掲載の例と、参考の最小例

公式ブログ掲載の例 公式

デスクトップ環境をバックグラウンド起動（ポート公開）:

PowerShell（Windows）
PS> wslc run -d --name=webtop -e PUID=1000 -e PGID=1000 -e TZ=Etc/UTC -p 3000:3000 -p 3001:3001 lscr.io/linuxserver/webtop:ubuntu-kde

GPU が使えるか確認（PyTorch + CUDA スクリプト）:

PowerShell（Windows）
PS> wslc run --rm --gpus all pytorch/pytorch:2.5.1-cuda12.4-cudnn9-runtime python -c "import torch; print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0))"

参考：Docker 風の最小起動 公式未掲載

wslc は Docker 風コマンド体系のため、次のような最小起動も可能と考えられます。これは公式ブログには載っていない参考例で、実際の挙動・対応フラグは wslc --help 等で確認してください。

PowerShell（Windows・参考）
PS> wslc run --rm -it ubuntu:latest bash
💡
公式の例で確認できるフラグ: -d（バックグラウンド）・--name・-e（環境変数）・-p（ポート）・--rm・--gpus。Docker と同じ感覚で使えますが、完全な互換は公式に明言されていません。

9 未確認・注意点（プレビュー品質）

採用判断の前に確認しておきたいこと

  • pre-release 限定 / GA 前: 本番運用は非推奨。仕様は 2026秋の GA までに変更され得ます。
  • Windows バージョン要件: 公式ブログでは明記なし（各種報道は Windows 11）。Windows 10 等での可否は要確認。
  • Docker Compose 対応: 公式発表に記述なし。複数コンテナ構成を多用するなら要確認。
  • 性能の実測: virtiofs「2倍速」は公式値。自分のワークロードでの第三者ベンチマークは未公開。
  • エンタープライズ統合の提供状況: コンテナ向け Defender for Endpoint 監視は private preview、Intune ダッシュボードの公式対応は数週間以内（現状は GPO/ADMX）。MDE プラグインは ARM64 非対応・短命コンテナ未表示等の制限あり（§5）。
  • virtiofs の既知の課題: Windows バインドマウント上で非 root ユーザーが作成したファイルが root 所有として見え、書き込み再オープンできない等の不具合報告あり（microsoft/WSL #40719）。プレビュー段階の挙動として留意。
🛑
このマシンへの影響: 手順 2 は WSL を pre-release ビルドに更新します。安定版を保ちたい場合は、検証用マシン／後で安定版へ戻せる前提で実施してください。
